TICKET-059 — R5 (Timeline slice)

Journal page — Timeline tab, read-only event feed

Open Priority: Medium

Summary

Ship the Journal page as a read-only Timeline of every event across every plant the user owns. Backend aggregates from existing tables (care logs, photos, achievements, plant acquisitions) via a query object — no new persisted journal_entries table. Frontend renders the unified feed with sticky day-group headers, filter toolbar (plant + type + date range), and cursor-paginated infinite scroll.

In one sentence: turn the existing event-shaped data scattered across CareLog / PlantPhoto / Plant / Achievement into one chronological page Rob can read like a diary.

What you're building

Six artifacts, three backend + three frontend.

Model (query) api/app/models/journal_stream.rb
Read-only query object. Takes a user + filter params (plant_id, kind, before, limit) and returns an array of polymorphic event hashes, newest first. Pulls from CareLog, PlantPhoto, Plant (acquisitions), Achievement via four scoped queries, then merges + sorts by occurred_at + applies the cursor cap. Not an ActiveRecord model — plain Ruby class.
Controller api/app/controllers/api/v1/journal_controller.rb
Standard index action. Parses query params via params.expect, hands them to JournalStream.new(current_user, ...), renders { entries: [...], next_cursor: "..." }. Scope is already enforced inside the query object via current_user.plants.
Routes api/config/routes.rb
Add get "/journal", to: "journal#index" inside the existing api/v1 namespace.
Hook client/src/hooks/useJournal.js
Wraps TanStack Query's useInfiniteQuery. Accepts { plantId, kind, dateFrom, dateTo }. Returns the standard infinite-query shape — data.pages, fetchNextPage, hasNextPage, isFetchingNextPage. Query key follows the project rule: ['journal', filters].
Page client/src/pages/Journal.jsx
Replaces the /journal placeholder route. Renders the page header, FilterToolbar (plant chip + type chips + date-range), the timeline list (day-grouped, sticky day headers), and an empty/loading/error state when the feed is blank or fetch fails. No tabs UI yet — that lands when Photos + Milestones tabs follow.
Component client/src/components/journal/JournalEntry.jsx
Renders one entry. Avatar (or icon disc for system events), body line, time label, optional thumbnail for photo entries. Variant via entry.kind — colour accent + icon swap. No interaction — read-only.

Requirements

  1. Unified feed. The /journal page shows one merged chronological stream of every care log, every photo, every plant acquisition, and every achievement the current user owns. Other users' events never appear.
  2. Day grouping. Entries are grouped under sticky day headers (Today, Yesterday, then dated). Header shows the date and a "N entries" count.
  3. Filter — plant. A filter row above the feed lets the user narrow to a single plant. Active filter renders as a dismissible chip with the plant avatar + name + ×.
  4. Filter — type. User can narrow to one or more event kinds (water, feed, photo, achievement, acquisition). Active kinds render as dismissible chips.
  5. Filter — date range. User can pick a date range (preset shortcuts: Last 7 days / Last 30 days / Last 90 days / All time; plus a custom range picker). Active range renders as a dismissible chip.
  6. Clear all. A "Clear all" link removes every active filter at once.
  7. Pagination. Page loads the first batch (default 30). Scrolling near the bottom triggers the next batch via cursor — no Load-more button, no jump-to-page. Cursor is the occurred_at ISO timestamp of the last seen entry.
  8. Empty states. Brand-new user with no events sees an EmptyState card ("No events yet — log some care to see them here"). User whose filters exclude everything sees a different EmptyState ("Nothing matches these filters").
  9. Error state. If the API fails, the page shows a WidgetError or ErrorState (per TICKET-052) with a retry button. Doesn't blank-screen.
  10. Read-only. No CTA to create an entry. No edit, no delete. The page is purely a viewer.

Acceptance criteria

Visiting /journal after logging in shows the Timeline with at least one entry (the user's first care log or achievement).
Day headers stick to the top of the scroll container as you scroll past them.
Picking a plant in the filter row immediately narrows the feed and shows the plant chip in the toolbar.
Selecting Water + Feed in the type filter shows ONLY care log entries. Selecting Photo shows only photo entries. Combining type + plant works (AND, not OR).
Picking Last 7 days in the date filter caps the feed to entries from the last 7 days. Custom range picker accepts a from + to date.
DevTools → Network: scrolling past the first 30 entries fires another GET /api/v1/journal?...&before=... request. Cursor is the last entry's occurred_at.
DevTools → Network: GET /api/v1/journal returns scoped data only. Creating a second user via ./scripts/console.sh and curling with their token returns ONLY their events.
Brand-new user with zero events sees the empty-state card, not a blank scroll area.
Killing the API mid-fetch (docker compose stop api) shows the error state with a retry button. Bringing the API back + clicking retry recovers.
Keyboard tab order walks through filter chips → clear-all → entries cleanly. Sticky day headers don't trap focus.
Lint + tests pass (./scripts/lint.sh && ./scripts/run_tests.sh).

React concepts worth looking up

useInfiniteQuery (TanStack Query)
Newer than the useQuery shape you've seen. Same idea but the data is split into pages — each call to fetchNextPage() requests the next batch and appends it. You give it a getNextPageParam function that reads the previous page's response and returns the cursor for the next one (or undefined when there's no more). Rails analogy: it's kaminari's .next_page chain, except the cache holds every page you've ever loaded so back-scroll is instant.
Cursor pagination vs page-number
A polymorphic feed merges rows from four tables, so their IDs aren't monotonic and OFFSET N is meaningless. Cursor pagination uses a stable sort key (here occurred_at) and "give me rows older than X" semantics. Stable under concurrent writes — new entries never shift the page boundaries. The cursor is just the timestamp of the last entry seen; the server returns rows strictly older than it.
Intersection Observer for "load more on scroll"
Don't listen to scroll events — that fires constantly and is expensive. Drop a sentinel <div> at the bottom of the list, observe it via IntersectionObserver, and when it intersects the viewport call fetchNextPage(). There's already an example in Plant.jsx for the sticky-header pattern (sentinelRef) — same idea.
Sticky day headers — pure CSS
No JS. Each day group is a <section>; its header is position: sticky; top: 0. The scroll container needs to be the actual scroll context (overflow + height) and not a descendant — the sticky reference is the nearest scrollable ancestor. Watch out for parents with overflow: hidden — they kill stickiness silently.
Filter state lives in URL search params
Use useSearchParams from react-router. Filter chips read from searchParams.get('plant_id') and write via setSearchParams(...). This makes the filtered view shareable, back-button-friendly, and survives refresh. Same pattern House.jsx uses for view mode + space_id today.
Query keys are arrays — and changing them = new request
Project rule (CLAUDE.md): TanStack Query keys are flat tuples like ['journal', { plantId, kind, dateFrom, dateTo }]. When the filter object changes, the key changes, TanStack treats it as a new query, fires a new request, and caches each filter combo independently. Switching back to a prior filter is instant from cache.

Notes & gotchas

  • Don't introduce a journal_entries table. The plan flagged this explicitly — initial preference is the read-only aggregation query. Persisted journal entries become useful when users author notes; that's not in this ticket.
  • Polymorphic IDs aren't unique. Two events from different tables can both have id = 5. Use "#{kind}-#{id}" as the React key, never the raw id.
  • Sort all four sources by occurred_at, not created_at. CareLog's performed_at, PlantPhoto's taken_at (or created_at if absent), Plant's acquired_at, Achievement's earned_at. Normalise into one occurred_at key in the query object.
  • Achievements may not have plant_id. The login-streak achievement is on the user, not a plant. Decide: hide them when the plant filter is active, or show them as user-scope events with no avatar. Recommend hide-when-filtered.
  • Don't query for everything then filter in Ruby. Apply plant_id, kind, before, and date filters at the SQL level on each source query, then merge the (already small) result sets. Avoid loading the whole event history into memory.
  • Initial page size 30. Tune later if scroll feels janky. 30 is enough for an above-the-fold first paint without overloading the first request.
  • Filter UI uses existing primitives. Tile chips (the Tile size="chip" shape used in StartJungleDialog), DateInput from components/form/, ActionIcon for chip dismiss. Don't roll new primitives.
  • When you're done: run ./scripts/lint.sh && ./scripts/run_tests.sh, commit on a feature branch ticket-059-journal-timeline, then open a PR with feat(v2): prefix.

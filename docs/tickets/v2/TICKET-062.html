TICKET-062

Journal Calendar tab

Open Priority: Medium

Summary

Add the journal's third tab — Calendar — a month grid that plots what happened on each day: coloured dots for the kinds of events logged (water, feed, photo, milestone), a "today" marker, and a clear at-a-glance sense of activity. It sits beside Timeline and Photos in the same FileTabs the journal already uses.

This is the achievable core of the mockup's calendar. The mockup also shows weather pills and forward-looking care-schedule overlays — both depend on backend that doesn't exist yet, so they're explicitly deferred (see the deferred panel). Start with a read-only month view of past events.

In one sentence: a month-grid calendar tab in the journal that dots each day with the events logged on it.

Requirements

  1. A third tab, "Calendar", appears in the journal (on both /journal and the Plant Detail journal) and renders a month grid when selected.
  2. The grid shows the current month by default, with weekday column headers and a full 6-week grid; days spilling in from the previous/next month are visually muted.
  3. Today's cell is clearly marked.
  4. Each day that has journal events shows small coloured dots indicating which kinds occurred (water, feed, photo, milestone/achievement) — colour-coded consistently with the rest of the journal.
  5. The user can page to the previous/next month; an empty month renders an empty grid (no events), not a broken or blank panel.
  6. When the journal is scoped to one plant (Plant Detail), the calendar shows only that plant's events; the global journal respects the active plant/date filters where they make sense.
  7. The view is read-only — no creating or editing events from the calendar in this ticket.

Acceptance criteria

The journal shows three tabs; clicking "Calendar" renders a month grid, and switching back to Timeline/Photos still works.
A day on which you logged a watering shows a water-coloured dot; a day with a photo shows a photo-coloured dot; a day with several kinds shows several dots.
Today's cell is visually distinct from the rest.
Paging to a month with no events shows a clean empty grid; paging back returns to the populated month.
On a plant's Journal segment, the calendar dots reflect only that plant.
No weather pills and no future-dated "due" markers appear (those are deferred) — the calendar only reflects events that actually happened.
Lint and tests pass (the grid helper has unit tests; the tab has a Playwright check that it renders and pages months).

Deferred — do NOT build here

Weather pills. Open-Meteo is wired — api/app/clients/open_meteo_client.rb + GET /api/v1/weather — but only as a 7-day forward forecast (current conditions + next 7 days); there's no historical/archive fetch. A month calendar is mostly past days, and past-day pills need Open-Meteo's archive API, which isn't wired. So weather is a fast-follow, not part of this ticket: forward-forecast pills (the next ~7 cells) are technically possible today, but the historical half needs the archive endpoint added first. Don't stub fake weather; leave the slot out here.
Forward-looking schedule / "overdue" dots. Showing upcoming care due (and red overdue markers) means surfacing each plant's computed schedule onto future calendar cells. That's a meaningful piece tied to the scheduling model and deserves its own ticket. This calendar reflects the past (logged events) only.
Week view + day drill-down. The Week/Month toggle and tapping a day to filter the Timeline to that date are nice follow-ups, but month-only + read-only keeps this ticket shippable. Leave hooks compatible (don't hard-code "month" so deeply that a week view can't slot in later).

Open question to resolve first

Where does the per-day data come from? Two options: (a) reuse useJournal with date_from/date_to set to the visible month and group the returned entries by day client-side (cheapest — but the timeline's 30-item pagination means a busy month could be truncated, so you'd need to fetch the whole month, not paginate); or (b) a small dedicated read returning a compact day→kinds map for a month range (e.g. GET /api/v1/journal/calendar?month=… off JournalStream), which is pagination-proof and sends far less data. Recommendation: (b) if a month realistically exceeds ~30 events; otherwise (a). Decide based on expected volume before wiring the hook.

React concepts worth looking up

Deriving the grid during render, not in an effect. The 6×7 cell array is a pure function of (visible month, day→events map). Compute it inline (or via a useMemo only if you measure a real cost) — don't stuff it into useState and sync it with useEffect. State-mirrored-from-props is the classic source of "calendar shows last month for one frame" bugs.
Local UI state vs server state. The "which month am I viewing" is pure client UI state — a useState holding a date. The events for that month are server state — a TanStack query keyed by the month (and plant/filters). Keep them separate: changing the month changes the query key, which fetches that month. Don't try to hold fetched events in useState.
Query keys that include the month. Whatever hook you use, the visible month must be part of the query key (e.g. ['journal','calendar', month, filters]) so paging months caches each month independently and refetches correctly — same pattern as the filter-aware keys already used for the timeline and photos.
When NOT to memoise. A 42-cell array build is cheap; don't reach for useMemo reflexively (Vercel's "simple expression in memo" rule). Reserve it for if profiling shows the grid build is actually hot.
Date handling without a library. The project formats dates with the platform Intl/toLocaleDateString (see journalGrouping.js). Build the grid with plain Date math; mind month boundaries and that JS months are 0-indexed. No new date library unless there's a strong reason.

Notes & gotchas

  • Don't introduce empty tab shells. The journal's rule (from 059/060) is tabs render only when they have real content. Calendar now qualifies — but keep that bar in mind if tempted to stub the deferred bits.
  • Past, not future. This calendar reflects logged events. No "due tomorrow" / overdue overlay — that's the deferred schedule ticket. Including it accidentally will make the calendar look wrong the moment a plant's schedule shifts.
  • No fake weather. Leave the weather slot out entirely until the Open-Meteo backend lands.
  • Month math. JS Date months are 0-indexed and roll over; build the grid carefully and unit-test the boundaries (a month starting on Sunday, a month needing 6 rows, leading/trailing padding).
  • Pagination trap. If you reuse useJournal, remember it caps at 30 entries/page — a busy month would be truncated. Fetch the whole month or use the dedicated aggregation (open question).
  • Keep it shippable. Month-only, read-only. Week view + day drill-down are deliberate follow-ups; don't scope-creep them in.
  • When you're done: run ./scripts/lint.sh && ./scripts/run_tests.sh, commit on a feature branch, then open a PR.

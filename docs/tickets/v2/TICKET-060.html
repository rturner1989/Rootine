TICKET-060 — R5 (Photos slice + tab chrome)

Journal Photos tab + FileTabs adoption, shared across /journal and Plant Detail

Open Priority: Medium

Summary

TICKET-059 shipped the Journal as a full-screen Timeline. This ticket adds the second tab — Photos — and with two tabs now real, promotes the page to the manila-folder FileTabs chrome the mockup always intended. The Photos tab is a date-ordered masonry grid of every plant photo the user owns, with a fullscreen lightbox, delete, and an in-tab upload entry point.

Crucially, the whole journal (tab header + Timeline + Photos) becomes one shared component rendered in two places: the global /journal page (all plants) and the Plant Detail page's Journal view (scoped to one plant). Same code, two scopes.

In one sentence: turn the standalone Timeline into a tabbed Journal with a Photos grid, and make that tabbed Journal work both globally and per-plant from a single component.

What came before: TICKET-059 (PR #65) built the Timeline as the whole page — read-only, day-grouped, filterable, infinite-scroll. It deliberately shipped without tab chrome, because a single-tab folder is pointless. It also left two reusable assets behind: the JournalStream query object (which already emits photo entries), and the Popover primitive. Separately, commit 0668b71 added PlantPhoto + a working upload flow at /plants/:id/photos/new.

What this unlocks: once two tabs justify the FileTabs header, the Milestones tab (computed plant anniversaries — no DB table) and the Schedule tab (month calendar + weather, Phase 2) slot in as further panels with no structural rework. This ticket also retires the Plant Detail JournalView stub by pointing it at the real shared component.

Mental model: think of the shared Journal component as a Rails partial rendered with a local — render "journal", scope: :all on the global page and render "journal", scope: @plant on the plant page. Same partial, the local decides how wide the net is. Here the "local" is an optional plantId prop that, when present, locks the feed to one plant and hides the plant filter.

What you're building

One shared journal component, a Photos tab with its grid + lightbox, a thin data hook, and the wiring that makes both the global page and the plant page render the shared component. Backend is mostly already done — the open question is whether Photos needs its own endpoint.

Requirements

  1. Tabbed journal. /journal shows a FileTabs header with exactly two tabs — Timeline and Photos. Timeline is the default. The existing Timeline feed (day groups, filters, infinite scroll) renders unchanged inside its panel.
  2. Photos grid. The Photos tab shows every photo the user owns as a date-ordered masonry grid, newest first. Photos from all the user's plants appear; other users' photos never do.
  3. Lightbox. Clicking a photo opens a fullscreen overlay with the full-resolution image, its caption, the plant's name, and the date taken. Escape, backdrop click, and a close button all dismiss it. Focus is trapped while open and returns to the clicked tile on close.
  4. Delete. A photo can be deleted from the grid (or lightbox), gated behind a confirm step. After deletion the photo disappears from the grid immediately, and the Timeline no longer shows it either (no stale duplicate).
  5. Upload from tab. The Photos tab has an "Add photo" call-to-action. It can route to the existing upload flow or open an inline uploader — either way, a freshly uploaded photo appears in the grid without a manual refresh.
  6. Shared, scoped journal. The Plant Detail page's Journal segment renders the same tabbed journal, scoped to that one plant. In this scoped mode the plant filter is hidden (filtering one plant's journal by plant is meaningless); type + date filters remain.
  7. Empty + error states. A user with no photos sees a Photos empty state distinct from the Timeline's. A failed photos fetch shows an error state with retry, never a blank panel.
  8. Keyboard + tab semantics. The tab header is keyboard-navigable (arrow keys between tabs — FileTabs already does this). The grid is reachable by keyboard; the lightbox is a proper modal.

Acceptance criteria

Visiting /journal shows a two-tab folder (Timeline | Photos). Timeline is selected by default and looks identical to what shipped in 059.
Clicking the Photos tab shows a masonry grid of photos, newest first. Arrow keys move between the two tabs.
Clicking a photo opens a fullscreen lightbox with caption + plant name + date. Escape closes it and focus lands back on the tile you clicked.
Deleting a photo (with confirm) removes it from the grid immediately. Switching to the Timeline tab shows it's gone there too — no stale copy.
Uploading a photo via the Photos-tab CTA results in the new photo appearing in the grid without a manual page refresh.
Opening a plant's detail page → Journal segment shows the same two-tab folder, but only that plant's events/photos, and the Plants filter is absent from the toolbar.
DevTools → Network: the photos request returns only the current user's photos. A second user (via ./scripts/console.sh + their token) sees only theirs.
A user with zero photos sees the Photos empty state, not a blank grid. Killing the API mid-fetch shows an error state with retry.
Reduced-motion: lightbox open/close + tab swap degrade to no/!minimal motion.
Reviewed via /dhh-rails-reviewer (if backend changes) + /react-best-practices + /accessibility; blocking findings addressed.
Lint + tests pass (./scripts/lint.sh && ./scripts/run_tests.sh).

React concepts worth looking up

Lifting the journal into one shared component
The Timeline currently lives directly in Journal.jsx. Moving it into JournalTabs so two pages can render it is "lifting" — the same instinct as extracting a Rails partial so two views can render it. The page becomes a thin shell; the reusable guts live in the shared component. The plantId prop is the partial's local.
Reusing FileTabs you already have
FileTabs (built earlier) owns the tab header, roving tabindex, and arrow-key navigation. You give it a tabs array + an activeId + an onChange; it renders the folder chrome and a single FileTabs.Panel slot. You don't re-implement tab a11y — you feed it data. Adding Milestones/Schedule later is appending to the array.
useInfiniteQuery, second time around
The Photos grid paginates the same way the Timeline does — useInfiniteQuery + an IntersectionObserver sentinel. You already have the pattern in useJournal + Journal.jsx. Reuse the shape: data.pages, fetchNextPage, hasNextPage, getNextPageParam reading a cursor. Query key gets a plantId in it so the global and per-plant grids cache separately.
Mutations + cache invalidation across two queries
Deleting a photo is a useMutation. The catch: photos live in two caches — the photos grid AND the journal Timeline (a deleted photo would still show as a Timeline entry). On success, invalidateQueries for both ['photos', …] and ['journal', …]. This is the Rails turbo_stream "refresh the things this change touched" instinct — except you name the keys explicitly. Per the project's mutation-cache rule, invalidateQueries is the default here (a delete is a 204 with cascading effects — nothing to optimistically patch).
The lightbox is a modal — trap focus, restore it
A fullscreen overlay must (1) move focus inside on open, (2) trap Tab within it, (3) close on Escape, (4) restore focus to the element that opened it. The Dialog primitive already solves all four — lean on it (or the same approach) rather than hand-rolling. The "restore focus to the tile you clicked" bit is the one people forget; it's the difference between a usable and a maddening keyboard experience.
CSS masonry without a library
A masonry layout (Pinterest-style staggered columns) is achievable with pure CSS columns — no JS, no library. Each tile is a column item; the browser flows them. Watch the gotcha: CSS columns fill top-to-bottom per column, so visual order runs down-then-across, not across-then-down. If strict newest-first reading order matters, that's the trade-off to weigh against pulling in a masonry lib. Default to CSS columns unless the order actually bites.

Notes & gotchas

  • Open decision — photos data source. Two routes: (a) reuse GET /api/v1/journal?kinds=photo and flatten the day-grouped pages client-side into a flat photo list, or (b) add a lean photos index endpoint (GET /api/v1/photos?plant_id=&before=&limit=) that returns flat, date-sorted photo rows. Recommendation: (b) — a masonry grid wants flat date-sorted rows, not day groups, and a dedicated endpoint keeps the payload lean (no need for the polymorphic entry envelope). It's a thin controller scoping through current_user.plants. But (a) is defensible if you'd rather not add API surface — your call, justify it in the PR.
  • Open decision — filters on Photos tab. The type filter is meaningless on Photos (everything is a photo). Decide: reuse JournalFilterToolbar with type hidden (plant + date only), or ship Photos v1 with no filters and add them when a volume signal appears. Lean: no filters on Photos v1 — keep it simple, the grid is already date-ordered.
  • Don't render Milestones or Schedule tabs. Two tabs only. No disabled shells. Same rule that kept Timeline tab-less in 059.
  • The <h1>Journal</h1> contract still holds. nav.spec.js asserts the /journal page's h1 is literally "Journal". The page shell keeps owning that heading; JournalTabs renders below it. Don't move the h1 into the tab component.
  • Delete touches two caches. Invalidate ['photos', …] AND ['journal', …] — a deleted photo is also a Timeline entry. Forgetting the journal key leaves a ghost photo entry in the Timeline until the next refetch.
  • Scoped mode hides the plant filter only. Type + date filters stay in per-plant mode. Don't strip the whole toolbar.
  • Reuse, don't re-roll. Lightbox focus management → borrow Dialog. Tab chrome → FileTabs. Confirm-before-delete → ConfirmDialog. Infinite scroll → the useJournal/Journal.jsx sentinel pattern. New primitives should be rare.
  • ActiveStorage URLs. PlantPhoto#image_url returns a rails_blob_url(only_path: true) path — already public after 059. Thumbnails vs full-res: the model serves one URL today; if grid perf bites, a variant is a follow-up, not this ticket.
  • When you're done: run ./scripts/lint.sh && ./scripts/run_tests.sh, commit on a feature branch ticket-060-journal-photos, then open a PR with the feat(v2): prefix.

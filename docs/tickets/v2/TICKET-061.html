TICKET-061

Journal stats rail — content

Open Priority: Medium

Summary

The journal's stats rail shipped as a working shell in TICKET-060: it collapses, persists, and shows a live total — but two of its three blocks say "Coming soon". This ticket fills them with real, filter-aware data: a per-kind breakdown, the most-active plants, and a streak.

All numbers must reflect the journal's active filters (plant / event-type / date range), exactly like the existing total does. The work is mostly backend (extending the existing JournalStream aggregation) plus swapping the placeholders for real markup.

In one sentence: turn the stats rail's placeholder blocks into real filter-aware stats — per-kind counts, top plants, and a streak.

Requirements

  1. The "In view" block shows, under its total, one row per event kind (Care, Photos, Achievements, Acquisitions) with a colour dot and that kind's count — and the counts reflect whatever filters are active.
  2. Kinds with a count of zero either read 0 or are omitted — decide what reads cleanest, but the block must never imply a kind exists when the filter excluded it (e.g. on the Photos-only view there is no Photos breakdown to show).
  3. The "Most active" block lists the plants with the most entries in the current filtered set, highest first, each with its avatar, nickname, and entry count. An empty result (e.g. a filter that matches nothing) shows a quiet empty line rather than a broken block.
  4. The "Streak" block shows a day count and a short hint line. The streak counts consecutive days ending today (or yesterday — decide in the open question) within the filtered set.
  5. Every number changes correctly when the user adds, removes, or clears a filter — the rail is never stale relative to the timeline beside it.
  6. The numbers are computed once on the server and read by the client; the client does not re-derive counts from the paginated entries it happens to have loaded.

Acceptance criteria

On /journal (desktop, rail open) with no filters, the rail's per-kind counts sum to the "In view" total.
Applying a "Photos" event-type filter makes the timeline show only photos AND the rail's total + breakdown update to match — no leftover Care/Achievement rows implying entries that are filtered out.
Applying a date filter (e.g. "Last 7 days") shrinks both the timeline and every rail number consistently.
"Most active" lists real plants from your data, ordered by count, with avatars — not placeholders.
Scrolling the timeline to load more pages does NOT change the rail numbers (they're whole-set totals, not "loaded so far").
A filter combination that matches nothing shows zeroed/empty rail blocks, not a crash or NaN.
Lint and tests pass (new model tests cover kind_counts, top_plants, and the streak).

Open question to resolve first

What is a journal "streak"? Two candidates: (a) consecutive calendar days that have any journal entry within the filtered set — self-contained, recomputed from the same data as the rest of the rail, and intuitive for a "journal" widget; or (b) reuse the existing care/login streak the app already tracks on the user — consistent with other surfaces but conceptually a different thing (and it wouldn't respect the journal's filters). Recommendation: (a), computed on JournalStream from distinct entry dates, so the rail stays a pure reflection of "what's in view". Confirm before building; it changes whether this is a new calculation or a field you read off current_user.

React concepts worth looking up

Reading a derived value from a TanStack infinite query. The rail already receives summary via data?.pages?.[0]?.summary — the server returns the same whole-set summary on every page, so page 0 is fine. Keep deriving the rail's values during render straight from that object; don't copy them into useState + a useEffect (that's the classic "derived state in an effect" anti-pattern — it just causes an extra render and risks going stale). If a value is a pure function of props/server data, compute it inline.
Server owns the calculation (project rule). The counts/top-plants/streak are business calculations, so they belong on the server (JournalStream) and ship via the summary — the client renders, it does not tally. Mirroring the counts client-side from the loaded pages would silently drift the moment pagination or filters change. This is the same rule that put days_until_water on the model rather than in the component.
Keys + list rendering. "Most active" and the per-kind rows are mapped lists — give each row a stable key (plant id, kind string). Index keys are fine only for a fixed, never-reordered list; these reorder as filters change, so use the real id/kind.

Notes & gotchas

  • Don't rebuild the rail mechanism. Toggle, collapse animation, the fixed-width-inner padding fix, localStorage, and desktop-only/global-only gating all shipped in 060. Touch only the block contents + the data feeding them.
  • Filters are the whole point. Reuse the existing per-source scopes so the new numbers carry the same plant/kind/date filters automatically. A count that ignores filters is the most likely bug here — test it.
  • The summary already ignores pagination + the cursor. Keep the new numbers consistent with that: they're whole-filtered-set totals, computed off the scopes without limit/before. (060's #summary tests show the pattern.)
  • "Most active" kinds with no plant. Achievements can be user-scope (no plant) and acquisitions are the plant itself — make sure the plant-id aggregation handles nil plants without polluting the top list (the existing feed_plant_ids already .compacts; mirror that).
  • Watch query count. This adds a handful of count/group queries per journal fetch. That's fine at current volume; if you reach for it, memoise within the request rather than pre-optimising.
  • Resolve the streak open question before coding it — it decides whether you write a new calculation or read an existing field.
  • When you're done: run ./scripts/lint.sh && ./scripts/run_tests.sh, commit on a feature branch, then open a PR.
